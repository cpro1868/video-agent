## SEC-001: API Key 泄露到 Git 历史

### 事件概要

| 项目 | 内容 |
|------|------|
| **事件编号** | SEC-001 |
| **发现日期** | 2026-06-25 |
| **处理日期** | 2026-06-26 |
| **严重程度** | 高危 (High) |
| **事件类型** | 敏感凭证泄露 |
| **影响范围** | Git 远程仓库历史、.claude 配置目录 |
| **状态** | ✅ 已闭环 |

### 事件详情

#### 泄露内容

| 文件 | 泄露类型 | 敏感信息 |
|------|---------|---------|
| `config.yaml`（根目录） | 提交到 Git 历史 | MiniMax API Key (`sk-cp-Q269R8...`) |
| `.claude/settings.local.json` | 提交到 Git 历史 | 包含相同 API Key 的 Bash 命令 |

#### 泄露根因

1. **开发阶段配置混用**：开发时直接在根目录创建 `config.yaml` 包含真实 API Key，用于本地测试
2. **.gitignore 配置滞后**：`.gitignore` 未及时配置，导致 `config.yaml` 和 `.claude/` 被提交
3. **权限配置未清理**：Claude Code 的 `settings.local.json` 中包含带 API Key 的 Bash 命令历史
4. **首次提交审查缺失**：first commit 前未进行敏感文件检查

#### 影响评估

| 维度 | 评估 |
|------|------|
| **机密性影响** | API Key 已暴露在公开仓库，任何人都可以获取 |
| **完整性影响** | Git 历史已被重写，协作分支需要强制同步 |
| **可用性影响** | 协作者需要重新 clone 或强制同步 |
| **业务影响** | Key 需要立即轮换，远程服务可能已被滥用 |
| **声誉影响** | 开源项目公开暴露凭证，影响用户信任 |

#### 处置措施

##### 1. 即时响应 (2026-06-26)

- [x] 停止使用泄露的 API Key
- [x] 在 MiniMax 平台轮换 API Key
- [x] 检查 API Key 使用日志，确认是否有异常调用

##### 2. 本地清理 (2026-06-26)

```bash
# 备份当前 .git 目录
cp -r .git /tmp/git_backup_20260626/

# 使用 git filter-branch 重写历史
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch config.yaml" \
  --prune-empty --tag-name-filter cat -- --all

git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch .claude/settings.local.json" \
  --prune-empty --tag-name-filter cat -- --all

# 清理 reflog 和不可达对象
git reflog expire --expire=now --all
git gc --prune=now --aggressive

# 强制推送到远程
git push origin --force main
```

- [x] 创建 `.githooks/pre-commit` hook，提交前扫描敏感模式

#### 经验教训

##### What Went Right

- 问题发现后立即启动响应流程
- 执行前创建了完整的 git 备份
- 分步验证清理结果，确保彻底
- 事后完善了防范机制，避免再次发生

##### What Went Wrong

- 开发初期未配置 `.gitignore`
- 首次 commit 前未进行敏感文件扫描
- 早期修复只删除文件，未清理历史
- Claude Code 权限配置未审核就提交

##### What We Learned

1. **防御性编程应包括版本控制防御**：敏感文件应在 commit 前就被检测，而非事后清理
2. **Git 历史一旦污染，清理代价极高**：需要重写历史、强制推送、协调协作者
3. **AI Agent 的配置目录也需审核**：`.claude/` 等隐藏目录可能包含敏感信息
4. **文档规范必须与技术措施结合**：仅有规范不够，需要 pre-commit hook 等自动检查
